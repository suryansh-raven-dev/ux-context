# Raven Design System Guidelines

**Version:** 1.0 (draft) &middot; **Owner:** Design (Suryansh)

**Audience:** Frontend engineers, future designers, LLM code/UI generators.

**Scope:** Raven Copilot (AI-SEARCH), Near Miss Management System (NMMS), P&amp;ID Tool.

**Foundation:** `raven-ui` — the versioned component library distributed as a package and rendered in this Storybook.

---

## How to use this document

These guidelines have three readers.

1. **Frontend engineers** building product surfaces. Read *Product Archetypes*, *Three-Tier Contract*, and the component docs in the rest of Storybook. Treat every rule with **MUST** as a PR-blocker. Treat **SHOULD** as the default unless you have a documented reason. Treat **MAY** as permission, not encouragement.

2. **Designers** extending any product. Read the whole *Guidelines* section end to end. The cross-product decisions that keep the system coherent live in *Product Archetypes* and *Three-Tier Contract*.

3. **LLMs** generating or editing Raven UI code. Load the *Product Archetypes* and *Three-Tier Contract* pages as system context before any UI generation request. When rules conflict, the narrower rule wins — product-specific over cross-product, pattern over component default.

Keywords in these docs follow **RFC 2119** conventions: MUST, MUST NOT, SHOULD, SHOULD NOT, MAY.
